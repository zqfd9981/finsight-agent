## 背景

FinSight V1 现在已经有三份核心治理文档：共享接口文档、并行开发规则文档和全局项目状态文档。这些文档已经把对象边界、模块分组、依赖关系和第一批阶段目标写出来了，但还没有把“多个窗口每天怎么开工、怎么同步进展、怎么收口”落到一套固定的项目结构里。

当前约束是：

- 项目还处在从 spec 走向真实实现的阶段，应用代码还没成形
- 后面希望开多个 Codex 窗口并行推进不同模块
- 这套文档必须够轻，不然维护文档本身就会拖慢开发
- 新窗口启动时读的内容必须少，不能每次都重新看完全部 spec

所以，这次不是加新的业务能力，而是在现有的 `parallel-delivery-governance` 基础上，补一套最小可用的并行开发模板。

## 目标和不做的事

**要达到的目标：**
- 为三大模块群提供固定位置的模块状态文件
- 为并行窗口提供任务卡、同步卡、交接卡三种模板
- 说明主控窗口和工作窗口各自更新什么
- 让新窗口只看共享接口、模块状态和任务模板就能快速接上

**这次不做的事：**
- 不创建新的业务代码目录或应用框架
- 不改共享接口里的字段定义
- 不把整个流程做成很重的项目管理系统
- 不要求每个小改动都生成完整交接记录

## 主要决定

### 1. 新增模板放在 `docs/finsight/` 下面，而不是放到 `openspec/specs/` 里

理由：
- 这些文件本来就是给日常开发用的，不是拿来描述系统运行时行为的
- 和现有的 `docs/finsight/shared-contracts-v1.md`、`project-status.md`、`parallel-delivery-governance.md` 放在一起，主控窗口查起来最顺手
- OpenSpec 里只保留“规则变了什么”，不把具体日常模板混进去

备选方案：
- 放在 `openspec/changes/...` 里当一次性工件。没有采用，因为这些模板不是只用一次，而是后面开发过程中一直要用

### 2. 模块状态文件按“模块群”来建，不按每个 spec 单独建

建议目录：
- `docs/finsight/modules/control-plane-status.md`
- `docs/finsight/modules/data-evidence-status.md`
- `docs/finsight/modules/presentation-eval-status.md`

理由：
- 现在项目里真正适合并行推进的就是三大模块群，这样分已经够用了
- 比按每个 spec 单独建文件更轻，不容易把文档写炸
- 和现有治理文档里的分组方式一致，不需要换脑子

备选方案：
- 每个 spec 一份状态文件。没有采用，因为当前阶段这样做太重，会增加窗口切换成本

### 3. 协作文档分成任务卡、同步卡、交接卡三种，而不是只保留交接卡

建议目录：
- `docs/finsight/templates/task-card.md`
- `docs/finsight/templates/sync-card.md`
- `docs/finsight/templates/handoff-card.md`

理由：
- 并行窗口最常发生的是派工和阶段同步，不是完整交接
- 任务卡回答“这轮做什么”，同步卡回答“现在做到哪”，交接卡回答“下一个窗口怎么接着做”
- 分开以后，每一种模板都可以很短，使用门槛更低

备选方案：
- 所有场景都用一张交接卡。没有采用，因为这样每次同步都要写太多内容，反而更累

### 4. 更新责任采用“主控窗口收口，工作窗口局部汇报”的方式

规则：
- 主控窗口负责更新 `project-status.md`
- 主控窗口负责在阶段检查后更新对应模块状态文件
- 工作窗口只需要维护和自己任务相关的任务卡、同步卡或交接卡

理由：
- 能避免多个窗口同时改同一份全局状态文档
- 能把全局结论集中到一个负责人手里，减少状态跑偏

备选方案：
- 所有窗口都可以直接更新全局状态。没有采用，因为这样冲突和口径不一致的风险更高

## 风险和取舍

- [风险] 模板过多，导致大家把时间花在维护文档而不是推进实现
  -> 处理办法：先限制为三份模块状态文件和三份模板，不再继续往下细分

- [风险] 模块状态文件与全局状态文档内容重复，造成口径漂移
  -> 处理办法：规定全局状态只记里程碑和卡点，模块状态只记本轮目标和模块内活跃任务

- [风险] 工作窗口不更新同步卡，主控窗口无法正确收口
  -> 处理办法：把阶段同步卡定义成并行窗口最基本的汇报动作，而且字段尽量少

- [风险] 项目后续规模扩大后，按模块群维护可能仍然过粗
  -> 处理办法：第一版先按模块群落地；如果后面某一组持续变大，再考虑拆成更细的状态文件

## 落地顺序

1. 在 `docs/finsight/` 下新增 `modules/` 与 `templates/` 目录
2. 创建三份模块状态文件，并填入第一版初始内容
3. 创建任务卡、同步卡、交接卡模板
4. 在 `parallel-delivery-governance.md` 中补充这些文件的角色说明与使用规则
5. 在 `project-status.md` 中补充“如何与模块状态文件协同更新”的说明
6. 用第一轮基于 mock 的并行开发试跑一下，看看这套模板够不够轻

## 还没定死的问题

- 三份模块状态文件要不要先写负责人名字，还是先保留“待分配”更合适
