## Purpose

定义 FinSight Agent V1 并行开发阶段的交付治理能力，包括模块分组、依赖关系、项目状态跟踪和联调检查点规则。

## 重点关注

- 现有 7 个 spec 如何按并行开发视角分组
- 哪些模块可以独立推进，哪些需要协同推进
- 项目状态和联调检查点如何被统一记录

## 非职责范围

- 不替代各 capability spec 的功能 requirement
- 不直接负责业务实现逻辑

## 上下游关系

- 上游输入：现有 7 个 capability spec、共享 contract 定义
- 下游输出：模块群分组、依赖图、项目状态文档、联调检查点

## Requirements

### Requirement: 并行交付治理定义模块分组与依赖图
项目 MUST 将现有 7 个 V1 spec 归入明确的并行交付模块群，并发布这些模块群之间的依赖关系，用于判断哪些工作可以独立推进、哪些需要协同检查点。

#### Scenario: 项目为并行实现分配 owner
- **WHEN** 项目开始为模块或团队分配 owner
- **THEN** 交付治理文档 MUST 标明每个 spec 所属的模块群，以及影响其联调顺序的上下游依赖

### Requirement: 项目状态文档跟踪真实交付状态
项目 MUST 维护一份统一的状态文档，记录按 spec 或按模块群的 owner、当前状态、依赖、blocker、里程碑和完成定义。

#### Scenario: 负责人查看当前推进状态
- **WHEN** 项目负责人或模块 owner 需要查看当前交付就绪度
- **THEN** 状态文档 MUST 直接展示各 spec 或模块群的当前进展与活跃 blocker，而不依赖各团队各自维护的私有笔记

### Requirement: 联调就绪检查点依赖 contract ready
项目 MUST 定义以 canonical contracts 和 mock payload 就绪为前提的联调检查点，使下游模块不必等待所有上游 live implementation 完成后才能启动。

#### Scenario: 下游模块在 live integration 前先行开发
- **WHEN** 某个下游团队已经拿到审批通过的共享 contract 和 mock payload
- **THEN** 交付治理文档 MUST 允许该团队先进入实现和本地验证，而不是等待上游 producer 的 live output 完全可用

### Requirement: 并行开发需要有模块进度文件
项目 MUST 为每个模块群维护独立的模块状态文件，使并行窗口在不重新阅读全部 spec 的情况下就能恢复当前模块的目标、依赖、输出和 blocker。

#### Scenario: 主控窗口准备开始新一轮并行开发
- **WHEN** 主控窗口准备为控制面、数据与证据面或呈现与评测面分配新的短期任务
- **THEN** 项目必须为对应模块群提供一份模块状态文件，其中至少包含模块范围、当前里程碑、本轮目标、冻结输入、本轮输出、活跃任务、卡点、禁止改动范围、下一次阶段检查和完成定义

#### Scenario: 新窗口恢复模块上下文
- **WHEN** 一个新的并行工作窗口开始负责某个模块群中的短期任务
- **THEN** 该窗口必须能够只通过共享接口文档、对应模块状态文件和必要的任务卡快速恢复上下文，而不必重新通读全部能力说明

### Requirement: 并行开发需要统一的窗口协作文档模板
项目 MUST 提供面向并行窗口的标准协作文档模板，用于任务分发、checkpoint 同步和窗口关闭时的交接。

#### Scenario: 主控窗口向并行窗口派发任务
- **WHEN** 主控窗口向某个并行工作窗口分配一项新的短期任务
- **THEN** 项目必须提供一份任务卡模板，至少包含任务名、目标、负责范围、禁止修改范围、输入依赖、预期输出和完成标准

#### Scenario: 阶段检查时收集并行进展
- **WHEN** 主控窗口到达一次约定的阶段检查并向各个并行窗口收集进展
- **THEN** 项目必须提供一份同步卡模板，至少包含已完成、当前卡点、是否影响其他窗口和下一步

#### Scenario: 工作窗口关闭或暂停
- **WHEN** 某个并行工作窗口准备关闭、暂停或把任务交给后续窗口继续
- **THEN** 项目必须提供一份交接卡模板，至少包含任务、当前结论、已修改文件、未决问题和下一步第一动作

### Requirement: 并行开发要明确谁更新哪些文件
项目 MUST 明确全局状态文档、模块状态文件和窗口协作文档之间的更新责任边界，避免重复记录和状态漂移。

#### Scenario: 阶段检查后统一收口项目状态
- **WHEN** 一轮并行开发结束并进入阶段检查后的收口
- **THEN** 主控窗口必须更新全局项目状态文档与相关模块状态文件，而不要求每个工作窗口直接修改全局状态文档

#### Scenario: 工作窗口仅需汇报局部进展
- **WHEN** 某个并行工作窗口只完成了局部对象或单个里程碑子项
- **THEN** 该窗口必须只更新或提交与自身任务直接相关的任务卡、同步卡或交接卡，而不应擅自改写其他模块状态或全局里程碑结论
