# 搜索质量提升与轻量 Rerank 方案说明

日期：2026-07-08  
状态：待实现  
文档类型：模块设计说明

## 1. 这次要解决什么

当前事件分析链路已经能跑通，但搜索质量明显不够稳，核心问题有四类：

- 外部搜索结果太薄，常常被 `top1` 噪音带偏。
- 外部新闻和官方披露结果没有经过足够的主题一致性筛选，容易出现 `topic mismatch`。
- 内部 RAG 虽然已经有 `rerank`，但当前只是字符重合打分，无法处理中文金融场景里“同词异题”的问题。
- 搜索结果召回后几乎直接被消费，缺少统一的“筛选后再进入分析”中间层。

这次工作主线只有两条：

1. 提升搜索质量  
2. 落一层轻量、可复用的 rerank 基础设施

不以“重做 planner/router”或“先换掉所有外部数据源”为目标。

## 2. 已确认的边界

- `if_weak` 逻辑已经在当前分支按既定思路调整，这次不改。
- 这次不写 spec，只保留一份面向实现的设计说明。
- 不做大而全的数据结构抽象，避免为统一而统一。
- rerank 直接优先采用免费模型，不先设计多级重排体系。

## 3. 关键设计判断

### 3.1 不过度设计统一结构体

外部新闻候选和内部 RAG chunk 本来就不是一种对象：

- 外部新闻天然有 `url/source/publish_date`
- RAG chunk 天然有 `company_name/doc_type/section_path`

如果硬把这些字段平铺到统一结构体里，会非常别扭，也没有必要。

这次统一的不是“业务对象”，而是 **rerank 所需的最小输入面**。

建议统一成一个很薄的输入接口：

- `id`
- `profile`
- `title`
- `text`
- `metadata`

其中：

- 外部新闻：
  - `title = 标题`
  - `text = 标题 + snippet`
  - `metadata = { url, source, publish_date }`
- 内部 RAG：
  - `title = 公司名/文档类型/段落路径 的简短组合`
  - `text = chunk_text`
  - `metadata = { company_name, company_code, doc_type, publish_date }`

也就是说，真正统一的是：

- query
- candidate text
- candidate short title
- 少量附加 metadata

而不是把所有候选都捏成同一种重结构体。

### 3.2 rerank 就是打分，接口应该尽量轻

这次不需要设计一套很重的领域对象系统。  
rerank 的本质就是：

- 输入一个 query
- 输入一批候选
- 返回每个候选的相关性分数
- 根据阈值保留或丢弃

因此建议接口直接做成类似 QA 对的形式。

输入：

- `query`
- `profile`
- `candidates[]`
  - `id`
  - `title`
  - `text`
  - `metadata`

输出：

- `results[]`
  - `id`
  - `score`
  - `keep`
  - `reason`（可选）

首版里真正必需的其实只有：

- `id`
- `score`

`keep` 和 `reason` 更多是为了 trace 和调试方便，可作为轻量扩展保留。

### 3.3 外部新闻链和内部 RAG 链共享同一个 rerank 接口

虽然候选对象不同，但两条链的核心流程是一样的：

1. 先召回一批候选
2. 做一些廉价过滤
3. 统一送进 rerank
4. 只保留高分结果

因此这次应复用的是：

- rerank 调用协议
- rerank 输出格式
- keep/drop 阈值逻辑

而不是强行统一所有上游候选字段。

## 4. rerank 模型选择

这次建议直接使用免费的中文/多语 rerank 模型，不先引入复杂分层设计。

首选：

- `BAAI/bge-reranker-v2-m3`

理由：

- 对中文场景友好
- 多语支持成熟
- 适合做通用相关性排序
- 体量和可用性相对平衡

备选：

- `Qwen/Qwen3-Reranker-0.6B`

理由：

- 体量更轻
- 中文能力也值得尝试
- 可作为低资源环境备选

现阶段不推荐再把方案扩成：

- 规则粗排一套
- rerank 模型一套
- LLM 精排一套

这种多层设计先不做，首版先把统一 rerank 中间层立起来。

## 5. 外部搜索链怎么改

这次不把 Bocha、巨潮、上交所、新浪、TianAPI 这类源当作“可直接信任的事实源”，而是统一降级为：

**候选召回源**

也就是说，外部源的职责是：

- 返回标题
- 返回 snippet
- 返回来源
- 返回时间
- 返回链接

然后由系统自己做后续筛选。

建议流程：

1. 外部源扩大量召回，不再只吃 `top1`
2. 先做规则过滤
   - 域名黑名单
   - 重复折叠
   - 明显广告/导流结果过滤
3. 把剩余候选转成轻量 rerank 输入
4. 用统一 rerank 模型打分
5. 只保留高分结果
6. 对少量高分结果抓正文
7. 再用 LLM 做正文摘要和 supporting points 提炼

注意：

- 正文抓取不是全量做
- 只对 rerank 后的前 `2~3` 条做

## 6. 内部 RAG 链怎么改

内部 RAG 当前已有：

- sparse retrieval
- dense retrieval
- fusion
- 一个很弱的字符 overlap rerank

这次不推翻前面的召回结构，重点只改最后一层重排：

- 保留 sparse/dense/fusion
- 把原来的弱 rerank 替换成统一 rerank 接口

这样可以直接解决：

- “钢铁去产能”和“生猪去产能”字符相似却不是同题
- “影响/催化/景气度”这类泛词导致的低质量 chunk 上浮

## 7. 为什么仍然需要 LLM

虽然 rerank 直接交给免费 rerank 模型，但 LLM 仍然是必要的，只是位置要放对。

这次我建议 LLM 继续负责两件事：

1. 对高分正文做摘要  
2. 提炼 supporting points / uncertainty notes

也就是说：

- 候选排序交给 rerank 模型
- 高质量正文压缩交给 LLM

这样比“让 LLM 直接给一堆候选逐条打分”更省 token，也更稳定。

## 8. 这次的最小实现目标

这次不追求一步到位，而是先做到下面这几点：

- 外部搜索不再只取单条结果直接进入分析
- 外部候选和内部 RAG 都接入统一 rerank 中间层
- 统一 rerank 输入输出保持轻量
- 先用免费 rerank 模型跑通
- 只对高分外部候选抓正文
- 抓正文后再交给 LLM 摘要
- 低分或 topic mismatch 结果不再继续硬答

## 9. 推荐实施顺序

1. 保留并透传外部原始候选列表
2. 扩大外部候选召回量
3. 增加规则过滤与去重
4. 落轻量 rerank 接口
5. 先把内部 RAG 接进统一 rerank
6. 再把外部新闻候选接进统一 rerank
7. 对外部高分候选增加正文抓取
8. 接 LLM 正文摘要
9. 补 trace 展示，方便调试 rerank 输入输出

## 10. 一句话总结

这次不是去设计一套复杂的搜索框架，而是把当前“搜到就信”的链路改成：

**多召回一点 -> 轻量过滤 -> 免费 rerank 模型打分 -> 只保留高分结果 -> 再抓正文并交给 LLM 摘要**

这样既能提升搜索质量，也不会把中间层设计得过重。
